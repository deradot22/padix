## Что работает хорошо (прод подтверждает)

- **Скамейка AMERICANA — образцово**: в 89% эвентов разброс сыгранных матчей между
  игроками = 0, ноль случаев «сидел 2+ раунда подряд» на все 64 эвента.
- **Ротация кортов**: прилипания нет (40.6% остаются на том же корте при ожидании 46.5%
  у случайного размещения).
- **Повторы партнёров**: 87% всех повторов математически вынужденные (длинные эвенты на
  8–12 игроках); реальная вина алгоритма — 60 повторов на 64 эвента, из них 22 даёт один
  дегенеративный эвент (см. BALANCED ниже).
- **Производительность**: полный перебор до 3 кортов с pruning — миллисекунды; на 4-5
  кортах жадный anchor-first — доли мс. Взрыва нет.

## Критические находки

### 1. MEXICANO: аутсайдер сидит на скамейке весь турнир (BUG)

`planMexicanoNextRound` (EventService.kt:446-462) берёт `standingsLeaderboard → SnakePairing.round().take(capacity)` —
за бортом остаются игроки с конца таблицы. Сидящий не набирает очков → снова последний →
снова сидит, до конца эвента. Тай-брейк (rating desc) тоже против слабого. Ротации
`playedRounds` (как в AMERICANA) в мексикано нет. Верхний лимит регистраций не
проверяется (`register` смотрит только рейтинг-гейт) — состав > courts×4 легко достижим.
**Сценарий: 5 игроков / 1 корт — пятый не сыграет ни одного матча.** Ломает суть формата.

### 2. FIXED_PAIRS: лишние пары молча выброшены из турнира (BUG)

`FixedPairsPairing.kt:15` — `teams.take(courtsCount*2)`: пары сверх вместимости не
получают bye, а вообще не попадают ни в один раунд, без ошибки. `registerPair` лимит пар
не проверяет, `startEvent` — только минимум. **Сценарий: 1 корт, 3 пары — третья пара
зарегистрирована и никогда не сыграет.** Плюс `dropLast(1)` при нечётном числе пар
выбрасывает последнюю пару из всего round-robin вместо стандартного bye (закреплено
тестом FixedPairsPairingTest.kt:44-48 как «норма»). Плюс пара №1 всегда на корте 1
(позиция circle method не вращается), второй круг после исчерпания RR невозможен (409).

### 3. BALANCED: связка cap=60 + превью ≠ план (BUG+DESIGN, подтверждено продом)

- **Дегенерация (прод: эвент 2026-05-17, 8 игроков, 7 раундов)**: жёсткий
  `BALANCED_TEAM_DIFF_CAP = 60` (EventService.kt:60) лексикографически доминирует над
  повторами партнёров → при широком спреде рейтингов единственный способ уложиться в cap —
  держать одни и те же пары «сильный+слабый». Итог: две пары не разлучались все 7 раундов
  (22 избежных повтора — треть всех избежных повторов прода в одном эвенте).
- **Превью ≠ план**: `previewBalancedRounds` и реальный `planSchedule` — два независимых
  НЕсидированных случайных прогона (`Random.Default`). Организатор подтверждает K
  «хороших» раундов в модалке, при старте может получить K′≠K (roundsPlanned молча
  перезаписывается, EventService.kt:586-588).
- При spread > ~120 `takeGoodRounds` даёт 0 → `coerceAtLeast(1)` создаёт единственный
  раунд, сам нарушающий cap.
- **Эмпирика: BALANCED почти не отличим от слепого ROUND_ROBIN** — медианный разрыв
  команд 124 Elo против 135 (−8%), p90 = 302 (оговорка: рейтинг в анализе текущий, не
  исторический). Все симуляционные тесты гоняют cap 150–250 — прод-значение 60 не покрыто.

### 4. Повторы соперников — систематическая слабость (DESIGN, подтверждено продом)

182 избежных сверх-встречи (3+ раза соперники при доступных свежих) в 27% эвентов.
Паттерн: 12 игроков × 12–17 раундов — партнёров алгоритм разводит, соперников нет
(пары соперников по 4-5 встреч). Причины: лексикографика (соперники оптимизируются
только при равенстве по партнёрам) + жадность по раундам без backtracking + чисто
жадный anchor-first на 4-5 кортах.

### 5. addFinalRound без гейтов (BUG)

EventService.kt:339-356: финальный раунд строится без проверки завершённости предыдущего
(в отличие от mexicano-ветки), а standings в POINTS-режиме подмешивают неподтверждённые
драфты (:375-376) → рассадка финала по искажённой таблице. Не гейтится по формату —
доступен для MEXICANO/FIXED_PAIRS прямым API-вызовом.

## Прочее

- DESIGN: скамейка AMERICANA при равных playedRounds всегда сажает слабейших
  (thenByDescending rating, PairingPlanner.kt:131-140) и выбирается ДО подбора пар,
  вне оптимизации. При N > 2×capacity слабейшие сидят два раунда подряд.
- DESIGN: addRound (AMERICANA) не требует завершённости предыдущего раунда — можно
  наплодить параллельных раундов; история при этом сидируется корректно.
- BUG (слабый): pruning в переборе некорректен на уровнях withinPenalty/tieBreak
  (отрицательный/пересчитываемый компонент, PairingPlanner.kt:184,203,287,296) — верхние
  уровни (партнёры/соперники/баланс) не страдают.
- MINOR: clearSchedule не чистит match_draft_scores (осиротевшие драфты после
  removePlayer→re-plan); race двух параллельных addRound → 500; OVERVIEW §9.2 описывает
  старый порядок BALANCED-компаратора; seed планировщика не логируется (жалобу на
  рассадку не воспроизвести).

## Не покрыто тестами (важное)

addRound+seedFromMatches (история при доп. раундах); вся цепочка Mexicano
standings→snake (включая bench-starvation); FIXED_PAIRS сверх courts×2 на уровне
EventService; greedy-путь 4-5 кортов; прод-cap BALANCED=60; addFinalRound.

## Рекомендации по приоритету

1. **MEXICANO**: ротация скамейки (учёт playedRounds, как в AMERICANA) + лимит
   регистраций по вместимости. Без этого формат нельзя рекламировать.
2. **FIXED_PAIRS**: 409 при регистрации пары сверх courts×2 + честный bye при нечётном
   числе пар + ротация кортов.
3. **BALANCED**: сделать cap мягким штрафом ниже partnerRepeats (вместо жёсткого
   доминирования), сидировать превью и план одним seed (и логировать его).
4. **Соперники**: поднять вес opponentRepeats (или объединить с partnerRepeats во
   взвешенную сумму вместо строгой лексикографики) — уберёт 4-5-кратные встречи на
   длинных эвентах.
5. addFinalRound: гейт на завершённость раунда + только AMERICANA + без драфтов в standings.

# padix · test-ui

Отдельный фронт для нагрузки/теста — создаёт пользователей и игры **через те же публичные API**, что и обычный `web/`. Никаких прямых записей в БД, никаких админ-эндпоинтов — всё как настоящий пользователь.

## Что умеет

### Страница «Пользователи»
- Регистрирует N пользователей через `POST /api/auth/register`.
- Сразу за регистрацией шлёт `POST /api/survey/submit` с разными `baseLevelCardId` (от новичка до турнирного) — чтобы получить разброс уровней.
- Сохраняет токен + `playerId` каждого юзера в `localStorage` для последующего использования.
- Кнопки на строке юзера: `/me` (проверить профиль), «Перелогин» (обновить токен), удалить из локального списка.

### Страница «Игры»
- Выбираешь организатора из созданных юзеров → используется его JWT.
- Заполняешь форму (дата, время, корты, scoringMode, очки) — данные те же, что просит `POST /api/events`.
- Регистрируешь либо первых N не-организаторов автоматически, либо выбираешь вручную чекбоксами.
- Каждый игрок регистрируется через `POST /api/events/{id}/register` (от имени организатора).

## Запуск (рекомендуется — Docker)

```bash
docker compose -f compose.dev.yml up -d test-ui
```

Открыть [http://localhost:8082](http://localhost:8082).

API дёргается через vite-прокси `/api → http://api:8080` (в docker-сети).

## Запуск локально (без docker)

```bash
cd test-ui
npm install
npm run dev
```

Откроется на [http://localhost:5174](http://localhost:5174). Прокси `/api` указывает на `http://localhost:8080` (или `API_TARGET` из env).

## Заметки

- Локальный стейт (список созданных юзеров) хранится в `localStorage` под ключом `padix-test-ui:v1`. Кнопка «Очистить локальный список» удаляет только локально — в БД пользователи остаются.
- Для калибровки нужно сыграть N матчей (по логике backend). Если хочешь стартовый рейтинг как у скриптов — расширь backend admin-эндпоинтом и добавь сюда кнопку. Прямой `UPDATE` в БД мы здесь не делаем намеренно — чтобы test-ui честно моделировал поведение пользователя.
